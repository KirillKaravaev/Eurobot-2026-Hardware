# EUROBOT 2026 — SCRAT Embedded Firmware (STM32H723)

Welcome to the embedded (low-level) firmware repository for the **SCRAT** robot of the **Engi-Teams** team for the **Eurobot 2026** competition.
The project implements low-level motion control, sensor fusion, and gripper management for the robot base, running on an **STM32H723VGT6** microcontroller (ARM Cortex-M7, 550 MHz).
Communication with the high-level PC stack is provided via **[micro-ROS](https://micro.ros.org/) over USB CDC (HS)**.

We also designed a PCB to accommodate the microcontroller board and conveniently connect various sensors and actuators to it, as shown below. We made every effort to use the available microcontroller peripherals as much as possible, so the board is quite versatile. Additionally, we designed a power board that complements the control board and provides the necessary voltages for powering sensors and actuators. By separating the power and control functions, we minimize interference during the control of powerful loads (such as motors and servos), while also reducing the overall size of the boards. ## Hardware

|---|---|
| MCU | STM32H723VGTx (ARM Cortex-M7, 550 MHz) |
| Drive motors | 3× JGB37 brushless (OMNI kinematics), impulse encoder feedback |
| Grippers | 4× ST3215 smart servo (IDs 1–4), UART4 @ 1 Mbaud |
| Arm alignment | 1× ST3215 servo (ID 5) — aligns boards during pickup |
| Height lift | 2× ST3215 servo (IDs 6–7) — synchronized lift mechanism |
| Odometry | PAA5160E1 (OTOS) optical sensor via I2C |
| Start button | GPIO input — enables motors and starts 100 s match timer |
| Debug output | UART7 — `printf()` routed via `_write()` |

| Watchdog | IWDG — resets MCU after 3 consecutive agent ping failures |
| Timing | DWT cycle counter — microsecond-precision interval measurement |

## micro-ROS topics

| Direction | Topic | Message type | QoS | Description |
|---|---|---|---|---|
| Subscribe | `twist_mux/cmd_vel` | `geometry_msgs/Twist` | Reliable | Velocity command (linear x/y, angular z) |
| Subscribe | `servo_spin` | `std_msgs/UInt8` | Reliable | Servo command byte (see table below) |
| Publish | `odom/unfiltered` | `geometry_msgs/Vector3Stamped` | Best effort | OTOS pose (x, y, θ), time-synced with ROS |
| Publish | `start` | `std_msgs/Bool` | Best effort | Start button state and 100 s match timer |

> Custom QoS profiles are used to separate reliable command delivery from best-effort sensor streams.

## Software architecture

The firmware runs two FreeRTOS tasks:

1. **Default task** — drives the micro-ROS executor at 50 Hz: reads OTOS, runs OMNI kinematics, publishes odometry and start state, handles incoming velocity and servo commands.
2. **Servo task** — polls `servo_spin_flag` set by the micro-ROS subscriber; executes blocking multi-step gripper sequences (delays up to ~3 s) without interfering with the micro-ROS loop.

The micro-ROS lifecycle is managed as a state machine:

```
WAITING_AGENT → AGENT_AVAILABLE → AGENT_CONNECTED → AGENT_DISCONNECTED → WAITING_AGENT
```

The watchdog is refreshed on every agent ping. After `MAX_PING_NO_RESPONSE_NUM` (3) consecutive failures the MCU resets automatically. While waiting for the agent, or after the 100 s match timer expires, all motors are stopped and OTOS tracking is reset.

## Servo commands (`servo_spin` topic, `std_msgs/UInt8`)

The command is a single byte. The **upper 4 bits** encode the command number; the **lower 4 bits** (`xxxx`) carry a per-command argument (gripper bitmask or grab-orientation mask). Bit 0 = gripper 1, bit 3 = gripper 4.

`servo_states[i]` is set by command **0–15** and remembered across calls — it encodes how each board was grabbed. Commands **80, 96, 112, 144, 240–255** read this state to decide in which direction to release each board.

| Command (bin) | Command (byte) | Action |
|:---:|:---:|---|
| `0000 xxxx` | 0–15 | **Pick up.** Descends, uses servo 5 to align boards during grab, closes all grippers, then rises to carry height. `xxxx` is saved as `servo_states[]` — encodes which boards need to be flipped when sliding off (used by bytes **80, 96, 112, 144, 240–255**). |
| `0001 0000` | 16 | **Calibrate height.** Sets the current lift position as the zero reference for height servos 6 and 7. |
| `0010 0000` | 32 | **Descend to mid.** Lowers the lift to an intermediate height. |
| `0011 0000` | 48 | **Home position.** Opens all grippers and returns the arm to the resting position. Does not reset the board memory from byte **0–15**. |
| `0100 0000` | 64 | **Rise to top.** Raises the lift to the top position. |
| `0101 0000` | 80 | **Slide 2 boards.** Slides all loaded boards off the top at once using the orientation from `servo_states` — each board flips over, changing its color. |
| `0110 0000` | 96 | **Stack 2 boards.** Places all loaded boards onto the surface at once using the orientation from `servo_states` — each board lands in its original orientation, preserving its color. |
| `0111 0000` | 112 | **Slide 1 board.** Slides the first board where `servo_states == 1` off the top — the board flips over and changes its color. All other loaded grippers stay closed. |
| `1000 0000` | 128 | **Release and reset.** Descends, opens all grippers unconditionally, then rises. Drops all boards without updating board memory. |
| `1001 0000` | 144 | **Stack 1 board.** Places the first board where `servo_states == 0` onto the surface in its original orientation, preserving its color. All other loaded grippers stay closed. |
| `1111 xxxx` | 240–255 | **Full stack sequence.** `xxxx` selects which grippers participate (one bit per gripper). Descends, repositions selected grippers according to their `servo_states`, rises, then releases. Non-selected loaded grippers are not affected. |

## Build & flash

The project is developed in **[STM32CubeIDE](https://www.st.com/en/development-tools/stm32cubeide.html)** and configured via `H723_Eurobot_optimized.ioc`.

**1. Clone with submodules:**
```bash
git clone --recurse-submodules <repo-url>
```

**2. Build** the `Debug` configuration in STM32CubeIDE.

**3. Flash** via ST-Link using the built-in IDE flasher, or with OpenOCD:
```bash
openocd -f interface/stlink.cfg -f target/stm32h7x.cfg \
        -c "program Debug/H723_Eurobot_optimized.elf verify reset exit"
```

**4. Start the micro-ROS agent** on the host PC:
```bash
ros2 run micro_ros_agent micro_ros_agent serial --dev /dev/ttyACM0 -b 115200
```

## How to set up micro-ROS

If you want to build the micro-ROS agent yourself or run it on a different STM32 microcontroller (yes, this is possible on virtually any microcontroller, not just the **[officially supported](https://micro.vulcanexus.org/docs/overview/hardware/)** ones), the following section may be helpful.

The full configuration instructions for micro ros are available in the official repository **[micro-ROS](https://github.com/micro-ROS/micro_ros_stm32cubemx_utils)** 
Here, I'd like to cover the topic of building the Micro Ros client separately. First of all, it's worth noting that ST produces a huge number of microcontrollers with different cores, unlike, for example, the Raspberry Pi Pico. Therefore, there is no pre-built library with the client. This is covered in the section **[Using this package with STM32CubeIDE](https://github.com/micro-ROS/micro_ros_stm32cubemx_utils#using-this-package-with-stm32cubeide)** The first build method, in which Docker is launched directly from STM32CubeIDE, was not possible to implement. This might be possible without much difficulty on Linux. For Windows 10/11, the second method may be helpful: manually building the container and then connecting the compiled library to the STM32CubeIDE project. 

When setting up the Transport configuration, we selected USB CDC, as our microcontroller supports USB 2.0 at speeds up to 12 Mbps. However, due to the agent-client interaction, as well as the specific operating characteristics of the USB library, the actual speed at this stage is far from the maximum, and this issue requires further study.

## Project readiness

The project is still under development, and therefore some functionality has not yet been implemented or is poorly implemented, with significant limitations and without a good description (which I plan to add in the future). The following modules, which were used in the Eurobot competition, are sufficiently implemented:

`kinematics` - used for calculation motor RPM from given velocities

`servo_ST3215` - used in gripper

`optic_odom_PAA5160E1` - used for optic odometry

`motor_jgb37_bl` - used for motor control

`odometry` - can be used to calculate velocities from measured RPM, but we used custom odometry message, because default message is too hight

## Known issues

**1. Loss of connection between the micro-ros agent and client** Cause: overloaded USB channel and incorrectly selected message priority. Solution: ping the agent several times.

**2. No response to gripper commands** Cause: incorrectly selected message priority (reliable by default), which could cause one message to overwrite another. The solution is to use a custom setting (qos - reliable to command messages and best effort to less important message stream from sensors) instead of the default ones with the same priority.

**3. Slow data publishing speed (about 50 Hz)** The probable cause is a suboptimal USB setting, causing slow data transfer. This issue is under investigation.
